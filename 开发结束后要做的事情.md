# 开发结束后要做的事情

> 本文档用于**明确区分「核心产品开发完成之后」才可以进行的事项**，
> 以防项目在早期阶段被带偏或过度扩展。

---

## 0️⃣ 文档定位（请先读）

- 本文档 **不参与任何核心功能决策**
- 本文档中的事项 **不得影响主工具的技术路线**
- 本文档内容 **只能在核心 exe 工具稳定后再执行**

核心产品始终是：
> **本地 · 单文件 · Windows `.exe` · 论文增量 Diff 工具**

---

## 1️⃣ 示例网页（Demo Web Page）

> ⚠️ 这是一个**说明性产物**，不是产品的一部分

### 1.1 目的（Why it exists）

示例网页仅用于：
- 直观展示「增量式、分块 diff」的效果
- 帮助潜在用户在 1 分钟内理解工具价值
- 为 README / 文档提供截图与静态示例

---

### 1.2 明确边界（Hard Boundaries）

- ❌ 不是 Web 版本
- ❌ 不提供在线对比功能
- ❌ 不与 exe 共享运行时代码
- ❌ 不作为未来产品形态的过渡

示例网页 **不改变、不影响、不替代** 核心工具。

---

### 1.3 技术范围（刻意受限）

```
- 静态 HTML
- 可选少量 CSS
- 可选少量纯前端 JS（仅用于展示）
```

禁止：
- 后端
- 框架（React / Vue 等）
- 构建工具

---

### 1.4 内容建议

示例网页可包含：
- 工具理念简述（Why incremental diff）
- 一小段「修改前 / 修改后」示例文本
- 对应的 diff 展示（截图或静态渲染）
- 工具界面截图

不包含：
- 实时交互
- 文件上传
- 在线处理

---

## 2️⃣ 文档与传播类事项

### 2.1 README 打磨

- 用非技术语言解释：
  - 这个工具解决什么问题
  - 它**不试图**解决什么问题
- 强调「exe / 本地 / 离线」

---

### 2.2 使用示例补充

- 增加 2~3 个真实论文修改场景示例
- 示例文本应偏学术风格
- 避免技术性 dummy 文本

---

## 3️⃣ 发布前检查（Release Checklist）

在对外发布前，确认以下事项：

- [ ] exe 可在无 Python 环境的 Windows 机器运行
- [ ] 对 UTF-8 / 常见论文编码兼容良好
- [ ] 未修改内容默认不展示
- [ ] 输出结果在长文本下仍然可读
- [ ] README 与示例网页表述一致

---

## 4️⃣ 明确不做的事情（再次强调）

以下事项即使有人提出，也应明确拒绝：

- 在线版本 / SaaS
- 多人协作
- 云存储
- AI 自动改写
- 复杂设置面板

---

## 5️⃣ 结语

这些事项的存在目的是：

- 帮助他人**理解项目**
- 而不是改变项目

如果某个“改进建议”需要修改核心设计，
那它就**不属于本文档的范畴**。

> **先把“哪里改了”这件事做到极致，其它的都可以慢慢来。**
